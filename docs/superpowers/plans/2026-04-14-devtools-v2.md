# DevTools v2 — Bottom Drawer Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox syntax for tracking.

**Goal:** Replace the 360px floating popover devtools with a full-width resizable bottom drawer that lets developers browse table/view data, trace mutations, and understand client vs server sync state.

**Architecture:** Bottom drawer in shadow DOM with 4 tabs (Data, Timeline, State, Actions). Template literals for rendering with event delegation. Client SQLite queries via BroadcastChannel devtools-query protocol. Server metrics from existing HTTP endpoint.

**Tech Stack:** Vanilla JS (template literals), CSS, BroadcastChannel API, SQLite (via data-worker)

## Task 1: Extend data-worker with devtools-query protocol and enriched status

The worker needs two additions: (1) handle devtools-query messages to run SQL against client SQLite, and (2) include schema table/view info + offline queue details in the devtools-status broadcast.

**Files:**
- Modify: packages/client/src/workers/data-worker.js

- [ ] **Step 1: Add devtools-query handler in the BroadcastChannel listener**

In data-worker.js, find the existing devtools BroadcastChannel message handler (around line 225). Add a new branch for devtools-query after the existing devtools-action handler:

```js
} else if (msg.type === 'devtools-query') {
    handleDevtoolsQuery(msg);
}
```

Then add the handler function after handleDevtoolsAction:

```js
function handleDevtoolsQuery(msg) {
    if (!db || !msg.id || !msg.sql) return;
    try {
        const rows = db.exec(msg.sql, { rowMode: 'object' });
        const columns = rows.length > 0 ? Object.keys(rows[0]) : [];
        devtoolsChannel.postMessage({
            type: 'devtools-query-result',
            id: msg.id,
            columns,
            rows,
        });
    } catch (e) {
        devtoolsChannel.postMessage({
            type: 'devtools-query-result',
            id: msg.id,
            columns: [],
            rows: [],
            error: e.message || String(e),
        });
    }
}
```

- [ ] **Step 2: Enrich devtools-status with schema info and offline queue entries**

In broadcastDevtoolsStatus (around line 272), extend the message payload. Replace the current entities field and views field with:

```js
tables: (_schemaTables || []).map(t => ({
    name: t.name,
    columns: t.columns || [],
    sql: t.sql,
})),
viewDefs: (_schemaViews || []).map(v => ({
    name: v.name,
    sourceTable: v.source_table,
})),
views: Object.assign({}, viewRowCounts),
offlineEntries: causalQueue.map(m => ({
    table: m._table || m.table || '',
    id: m._id || m.id || '',
})),
offlineQueue: causalQueue.length,
```

This replaces the hardcoded entities field. The tables and viewDefs fields carry schema metadata so the devtools sidebar can list tables/views without a separate query.

- [ ] **Step 3: Test manually**

Start the dev server from apps/test. Open the browser console on the app page. Verify the BroadcastChannel carries the new fields by sending a devtools-ping and checking the devtools-status response has tables and viewDefs arrays. Then test a query by sending a devtools-query message with sql SELECT * FROM products LIMIT 3 and confirming a devtools-query-result comes back with columns and rows.

Each card is a div with class se-action-card (plus variant class yellow/red), data-action attribute, optional data-confirm attribute with confirmation text, optional data-client-side attribute. Card content: label (bold) and description (muted).

Actions list:
- force-reconnect: default, client-side. Drop and re-establish the NATS connection.
- trigger-gc: default, server. Run garbage collection on the workspace stream.
- clear-client-db: yellow, client-side. Wipe local SQLite/OPFS and reload.
- purge-stream: yellow, server. Delete all messages from the NATS stream.
- teardown: red, confirm, server. Delete workspace and its stream.
- reset: red, confirm, server. Teardown + clear entity state + re-provision + clear client DB.

Update the executeAction event delegation handler to read data-action, check for data-confirm (show confirm dialog), and check data-client-side. The actual execute logic (parallel reset, BroadcastChannel, server POST) is copied from the existing implementation.
